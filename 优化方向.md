# 检修任务闭环与证据浏览优化设计（方案 B）

## 文档信息

| 项目 | 内容 |
|---|---|
| 文档状态 | 设计已确认，待实施规划 |
| 更新日期 | 2026-05-24 |
| 总体路线 | 业务同步、重任务异步混合型 |
| 第一阶段目标 | 完成检修任务闭环 MVP，并具备可追溯的手册证据浏览能力 |
| 设计范围 | Java 业务后端 + Python AI/RAG 模块的协作边界与接口支撑 |

## 1. 背景与目标

当前系统已有 Java 业务端、Python FastAPI/RAG 模块、Redis Stack 向量检索、MinIO 文件存储及 RabbitMQ 导入雏形。Python 已具备 PDF 文本、表格、图片解析和检索基础能力，Java 已具备手册管理及图谱相关服务基础。

现阶段的主要问题不是单一 AI 问答能力不足，而是尚未形成可信的业务闭环：

- 手册导入与 AI 引用之间缺少稳定版本和可用状态控制。
- 检修过程仍以会话或案例为中心，缺少任务、计划、审批、执行、验收的主线实体。
- 图谱验证未形成唯一权威路径，AI 结论与业务校验边界不够清晰。
- 已批准证据需要支持查看引用片段、表格结构和原 PDF，现有搜索接口不足以承担证据浏览。
- 现场图片、人工审批、高风险确认、案例入库等环节缺少统一审计链路。

本设计以“检修任务闭环 MVP”为第一阶段目标，实现以下业务链路：

```text
上传或更新手册
-> 异步解析并建立可引用索引
-> 创建检修任务并上传故障图片
-> AI 基于 ready 手册证据和 Java 图谱校验生成计划草案
-> 人工审批整份计划
-> 高风险步骤执行前再次确认
-> 上传执行证据并提交复测验收
-> 验收通过后生成案例草稿
-> 人工审核后写入正式图谱和知识索引
```

## 2. 已确认的设计决策

| 主题 | 决策 |
|---|---|
| 主数据归属 | 手册、任务、计划、步骤、审批、验收、案例状态均由 Java 管理 |
| AI 与 RAG | PDF 解析、向量索引、知识检索与结构化计划草案由 Python 管理 |
| 图谱诊断权威 | Java `GraphQueryService` 及对应内部 API 是唯一校验权威 |
| 手册索引 | Java 保存手册和文件稳定定位，异步触发 Python 建索引，仅 `ready` 服务版本可作为计划证据 |
| 任务入口 | 先创建任务，再显式请求生成计划草案 |
| 计划审批 | 整份计划先审批；Java 固定规则判定的高风险步骤执行前必须再次确认 |
| 风险判定 | Java 固定规则兜底，AI 仅提供风险提示，最终风险等级不得低于 Java 规则判定 |
| 检修等级 | 支持日常检修、故障维修、重大检修，并影响计划模板、审批及验收要求 |
| 案例沉淀 | 仅已验收任务生成案例草稿；仅审核通过案例进入图谱或正式知识索引 |
| 前端边界 | 新的检修闭环页面只调用 Java 业务 API；Python 现有搜索接口保留兼容，不作为新业务状态入口 |
| 降级规则 | 图谱暂不可用时，允许基于有效手册证据生成带警告草案，必须由审批人显式确认警告 |

## 4. 总体架构与职责边界

采用方案 B：耗时重任务异步处理，面向用户的关键业务交互保持同步可解释。

| 领域 | Java 后端职责 | Python AI/RAG 模块职责 |
|---|---|---|
| 维修手册 | 保存业务主数据、版本、MinIO 对象定位、服务版本、索引状态、原文访问授权 | PDF 解析、文本/表格/图片索引、分块统计和知识检索 |
| 图谱 | 管理设备、部件、故障、方案和案例关系，提供权威诊断与路径校验 | 调用 Java 内部接口，不独立维护诊断权威逻辑 |
| 检修任务 | 保存任务、计划、步骤、证据、审批、验收与案例审核状态 | 根据限定上下文生成诊断摘要、步骤草案、安全提示和验收建议 |
| 现场图片 | 保存图片引用及访问授权，向 Python 下发短期受控地址 | 使用图片参与检索或计划生成，不持有业务审批状态 |
| 案例沉淀 | 审核通过后创建正式案例及图谱关系 | 为已批准案例建立辅助知识表示或索引 |
| 对外接口 | 为新检修闭环前端提供统一业务 API | 保留知识服务能力及现有兼容接口，不直接改变业务状态 |

### 4.1 手册入库主流程

```text
Java 上传手册到私有文件存储
-> 保存 MaintenanceManual 与稳定文件定位
-> 创建 ManualIndexJob，候选状态 queued
-> RabbitMQ 发布 manualId、indexVersion 和文件定位
-> Python 解析 PDF 并构建 RAG 索引
-> Python 回传 ready / failed 及分块统计
-> Java 在成功时原子切换 servingIndexVersion
```

### 4.2 检修任务与计划主流程

```text
Java 创建 MaintenanceTask
-> 保存设备、故障描述、现场图片和检修等级
-> 用户请求生成计划
-> Java 筛选适用且 ready 的手册服务版本
-> Java 同步调用 Python 生成 PlanDraft
-> Python 仅检索 Java 下发范围内的手册证据
-> Python 调用 Java 图谱诊断与路径校验接口
-> Java 执行固定危险规则并保存计划版本
-> 人工审批整份计划
-> 任务进入待执行或执行中
```

### 4.3 执行、验收与案例主流程

```text
执行普通步骤并上传证据
-> 高风险步骤在执行前人工确认
-> 完成必要步骤
-> 提交复测数据与验收结论
-> 验收通过后任务完成
-> 生成 CaseRecordDraft
-> 人工修正并审核
-> 审核通过后写入正式案例、图谱关系和 Python 知识索引
```

## 5. Java 领域模型

| 实体 | 职责 | 核心字段或约束 |
|---|---|---|
| `MaintenanceManual` | 手册业务主数据 | `manualId`、名称、设备类型、手册类型、当前业务版本、`servingIndexVersion`、`candidateIndexVersion`、候选索引状态 |
| `ManualVersion` | 手册文件版本及访问定位 | `manualId`、版本、`storageBackend`、`sourceBucket`、`sourceObjectKey`、文件 MIME、上传时间 |
| `ManualIndexJob` | 一次解析和索引任务 | `jobId`、`manualId`、`indexVersion`、状态、分块统计、失败原因、请求追踪号 |
| `MaintenanceTask` | 检修业务主线 | 设备、故障描述、现场图片、检修等级、当前状态、创建人与创建时间 |
| `MaintenancePlan` | 某任务的一版计划 | `planId`、`taskId`、`revision`、生成请求号、审批状态、警告项、审批记录 |
| `MaintenanceStep` | 可执行步骤 | 顺序、准备或操作说明、Java 最终风险级别、AI 风险提示、执行确认要求、执行状态 |
| `PlanEvidence` | 计划固定引用的证据 | `manualId`、`indexVersion`、页码、`chunkId`、图谱路径引用及校验状态 |
| `ExecutionEvidence` | 现场执行证据 | 步骤、图片、测量数据、备注、提交人与时间 |
| `TaskAcceptance` | 复测与验收 | 验收项目、实测值、结论、附件、验收人及时间 |
| `CaseRecordDraft` | 待审核案例草稿 | 来源任务、内容、审核状态、审核意见及发布结果 |
| `AuditRecord` | 重要业务行为留痕 | 操作类型、目标实体、操作者、请求追踪号、前后状态、时间 |

AI 会话可以关联 `taskId`，但不能作为审批、步骤执行、验收或案例发布的状态依据。

## 6. 状态机与业务约束

### 6.1 手册索引版本切换

单一 `indexStatus` 无法表达“新版本失败但旧版本仍可用”。手册必须分别保存当前服务版本和候选索引状态：

```text
上传新版本
-> ManualIndexJob.QUEUED
-> ManualIndexJob.INDEXING
-> READY：原子切换 servingIndexVersion
-> FAILED：保留原 servingIndexVersion，记录候选失败原因
```

| 场景 | 处理规则 |
|---|---|
| 首次上传索引失败 | 手册不可作为计划证据 |
| 已有可用版本，新版本失败 | 旧服务版本继续可引用 |
| 历史计划引用手册后再次换版 | 历史计划继续固定引用原索引版本 |
| Python 返回分块统计 | Java 仅保存统计与状态，不复制向量内容 |
| 手册已停用或删除后收到迟到成功结果 | Java 不允许其切换为服务版本，并触发孤立索引清理流程 |
| 解析阶段发生异常 | Java 必须在发送索引任务前生成 `manualId`、版本与 `jobId`，使 Python 任一失败阶段都能稳定回写对应任务状态 |

### 6.2 检修任务状态

```text
CREATED
-> PLAN_GENERATING
-> PLAN_REVIEW_PENDING
-> APPROVED
-> IN_PROGRESS
-> ACCEPTANCE_PENDING
-> COMPLETED
```

可中止或失败状态：

```text
PLAN_GENERATION_FAILED
REJECTED
CANCELLED
```

| 状态 | 含义与约束 |
|---|---|
| `CREATED` | 已有任务记录，尚无可执行计划 |
| `PLAN_GENERATING` | Java 正在请求 Python 生成草案 |
| `PLAN_REVIEW_PENDING` | 有可审阅的计划版本和固定证据引用 |
| `APPROVED` | 整份计划已批准，尚未开始执行 |
| `IN_PROGRESS` | 至少一个步骤开始处理或已完成 |
| `ACCEPTANCE_PENDING` | 必要步骤完成，等待复测验收 |
| `COMPLETED` | 验收通过，任务闭环完成 |

约束：

- 没有适用 `ready` 手册证据时，任务可以创建，但不得形成可审批计划。
- Python 计划生成失败时，不生成空白或纯模型知识计划，任务进入可重试失败状态。
- 验收未通过时，任务保持在 `ACCEPTANCE_PENDING`，允许生成新的计划版本继续处理。

### 6.3 计划版本与审批

```text
draft -> pending -> approved
                  -> rejected
approved -> superseded（仅在新计划完成审批替换后）
```

| 场景 | 规则 |
|---|---|
| 计划批准前重新生成 | 建立新 `revision`，旧草案可标记为被替代 |
| 计划已批准但尚未执行，需调整内容 | 新建计划版本并重新整体审批 |
| 计划已开始执行 | 不允许原地修改步骤；现场变化时暂停并生成新版本 |
| 草案存在图谱降级警告 | 审批人必须显式确认警告后才能批准 |
| 同一任务 | 同一时间只能有一版已批准的可执行计划 |

### 6.4 高风险步骤确认

```text
普通步骤：PENDING -> READY_TO_EXECUTE -> COMPLETED
高风险步骤：PENDING -> WAITING_CONFIRMATION -> READY_TO_EXECUTE -> COMPLETED
```

高风险确认必须固定到 `taskId + planRevision + stepId`，并保存 `riskRuleCode`、确认人、确认时间和说明。计划版本改变后，旧确认不得复用。

### 6.5 验收与案例发布

```text
任务验收通过
-> MaintenanceTask.COMPLETED
-> CaseRecordDraft.PENDING_REVIEW
-> APPROVED：发布正式案例并触发图谱和知识索引
-> REJECTED：保留审核意见，不进入正式知识库
```

任务完成与案例发布相互解耦，避免未核验内容污染后续诊断依据。

## 7. 跨服务契约

### 7.1 接口类型

| 调用用途 | 调用方向 | 方式 |
|---|---|---|
| 手册解析索引 | Java -> Python -> Java | RabbitMQ 异步任务 + 状态回传 |
| 计划草案生成 | Java -> Python | 同步内部 API |
| 图谱诊断与路径校验 | Python -> Java | 同步内部 API |
| 已批准案例索引 | Java -> Python | 异步任务 |

### 7.2 手册索引任务消息

Java 发布的任务消息传稳定对象定位，不依赖会过期的预签名 URL：

```json
{
  "jobId": "idx_20260524_001",
  "manualId": "manual_001",
  "indexVersion": "v2",
  "file": {
    "storageBackend": "minio",
    "bucket": "manual-private",
    "objectKey": "manuals/manual_001/v2/source.pdf",
    "contentType": "application/pdf"
  },
  "metadata": {
    "manualName": "发动机检修手册",
    "equipmentType": "发动机",
    "manualType": "maintenance"
  },
  "requestedAt": "2026-05-24T17:30:00+08:00"
}
```

Python 成功结果：

```json
{
  "jobId": "idx_20260524_001",
  "manualId": "manual_001",
  "indexVersion": "v2",
  "status": "ready",
  "statistics": {
    "textChunks": 151,
    "tableChunks": 60,
    "imageChunks": 58,
    "imageSummaryChunks": 58
  },
  "completedAt": "2026-05-24T17:32:00+08:00"
}
```

Python 失败结果：

```json
{
  "jobId": "idx_20260524_001",
  "manualId": "manual_001",
  "indexVersion": "v2",
  "status": "failed",
  "errorCode": "PDF_PARSE_FAILED",
  "errorMessage": "PDF 页面解析失败",
  "completedAt": "2026-05-24T17:32:00+08:00"
}
```

一致性要求：

- Python 以 `jobId` 幂等处理重复消息。
- Java 仅允许当前候选版本的成功结果切换 `servingIndexVersion`。
- Python 建索引成功但状态回传失败时，支持状态回传重试或 Java 主动查询任务结果。
- 新索引失败不得删除旧的可用索引。

### 7.4 图谱路径校验调用

Python 在形成计划草案前调用 Java 内部接口：

```http
POST /internal/graph/diagnosis/validate
```

```json
{
  "taskId": "task_001",
  "equipmentType": "发动机",
  "faultDescription": "启动后出现异常振动与异响",
  "candidateFaults": ["轴承异常", "紧固部件松动"],
  "candidateSolutions": ["检查轴承磨损", "检查并重新紧固连接件"]
}
```

```json
{
  "validationId": "graph_check_001",
  "matches": [
    {
      "fault": "紧固部件松动",
      "solution": "检查并重新紧固连接件",
      "pathStatus": "verified",
      "pathId": "path_008"
    }
  ],
  "unverifiedItems": [
    {
      "fault": "轴承异常",
      "solution": "检查轴承磨损",
      "reason": "NO_GRAPH_PATH"
    }
  ]
}
```

只有真实存在的 `Fault -> Solution` 关系路径可标记为已验证。节点分别存在而无关联路径时，应标记为缺少图谱依据。

## 8. Java 对外业务 API

新的检修闭环前端以 Java 为统一入口，避免前端自行组合 Python 索引状态与 Java 审批状态。

### 8.1 手册与证据 API

| 接口 | 用途 |
|---|---|
| `POST /api/manuals` | 上传首次手册并创建索引任务 |
| `POST /api/manuals/{manualId}/versions` | 上传手册新版本并异步索引 |
| `GET /api/manuals` | 查询手册列表、服务版本和候选索引状态 |
| `GET /api/manuals/{manualId}` | 查询手册详情、版本及分块统计 |
| `GET /api/manuals/{manualId}/index-jobs/{jobId}` | 查询指定索引任务状态和失败原因 |
| `GET /api/manuals/{manualId}/versions/{version}/chunks` | 按类型和分页浏览证据分块 |
| `GET /api/manuals/{manualId}/versions/{version}/chunks/{chunkId}` | 查看单条分块与相邻上下文 |
| `GET /api/manuals/{manualId}/versions/{version}/source?page=12` | 获取受控原 PDF 访问能力并支持定位页码 |

上传或更新手册接口立即返回任务状态，不等待解析完成：

```json
{
  "manualId": "manual_001",
  "version": "v2",
  "indexJobId": "idx_20260524_001",
  "indexStatus": "queued",
  "servingIndexVersion": "v1"
}
```

### 8.2 任务与计划 API

| 接口 | 用途 |
|---|---|
| `POST /api/maintenance-tasks` | 创建检修任务 |
| `GET /api/maintenance-tasks/{taskId}` | 查询任务、计划、执行和验收概览 |
| `POST /api/maintenance-tasks/{taskId}/plans/generate` | 请求生成新版计划草案 |
| `GET /api/maintenance-tasks/{taskId}/plans/{planId}` | 查看计划步骤、证据、风险和警告 |
| `POST /api/maintenance-tasks/{taskId}/plans/{planId}/approve` | 批准整份计划 |
| `POST /api/maintenance-tasks/{taskId}/plans/{planId}/reject` | 驳回计划 |

创建任务请求：

```json
{
  "equipmentId": "equipment_001",
  "faultDescription": "启动后出现异常振动与异响",
  "maintenanceLevel": "FAULT_REPAIR",
  "imageIds": ["img_001"]
}
```

带图谱降级警告的计划生成响应：

```json
{
  "taskId": "task_001",
  "taskStatus": "PLAN_REVIEW_PENDING",
  "planId": "plan_001",
  "revision": 1,
  "approvalStatus": "pending",
  "warnings": [
    {
      "code": "GRAPH_VALIDATION_UNAVAILABLE",
      "message": "图谱校验暂不可用，计划仅基于有效手册证据生成，审批时需人工复核。"
    }
  ]
}
```

审批人批准带警告计划时必须提交已确认的警告代码，Java 不接受未确认警告的批准请求。

### 8.3 执行、验收和案例 API

| 接口 | 用途 |
|---|---|
| `GET /api/maintenance-tasks/{taskId}/plans/{planId}/steps` | 查看当前批准计划的步骤与执行状态 |
| `POST /api/maintenance-tasks/{taskId}/plans/{planId}/steps/{stepId}/confirm-risk` | 确认指定高风险步骤 |
| `POST /api/maintenance-tasks/{taskId}/plans/{planId}/steps/{stepId}/complete` | 提交步骤执行结果和证据 |
| `POST /api/maintenance-tasks/{taskId}/acceptance` | 提交复测验收 |
| `GET /api/case-drafts/{draftId}` | 查看由已验收任务生成的案例草稿 |
| `POST /api/case-drafts/{draftId}/approve` | 审核通过并触发正式发布 |
| `POST /api/case-drafts/{draftId}/reject` | 驳回案例草稿 |

## 9. 文档浏览与 RAG 证据支撑设计

原有优化文档提出的文档列表、详情、分块分页、上下文和原 PDF 查看能力纳入第一阶段，因为审批和验收需要核验 AI 引用依据。

### 9.1 对外边界与兼容策略

| 场景 | 规则 |
|---|---|
| 新检修闭环前端 | 只调用 Java `/api/manuals` 与任务相关接口 |
| Python 知识能力 | 为 Java 聚合接口提供解析、分块读取和检索支撑 |
| 既有 `POST /ai/knowledge/search` | 保持现有请求和响应行为可用，避免破坏已有调用 |
| 历史知识管理调用 | 可在迁移期保留 Python 现有路径，但不得越过 Java 修改手册业务状态、审批或案例发布 |

Java 手册实体是业务身份来源；Python 中用于向量与分块读取的 `document_id` 必须与 `manualId + indexVersion` 建立稳定映射，不能生成无法关联的独立文档身份。

### 9.2 文档列表与详情

Java 列表和详情响应需包含：

- `manualId`、业务名称、设备类型、手册类型。
- 当前 `servingIndexVersion` 与候选版本状态。
- 文本、表格、图片及图片摘要分块统计。
- 导入或索引失败原因。
- 稳定保存的创建时间和更新时间；不得仅依赖 Python 清单 Hash 外层存在但未返回的临时时间字段。
- 文件稳定定位是否完整、是否可打开原 PDF。

Python 如保留知识服务接口，其内部能力可对应：

```http
GET /ai/knowledge/documents
GET /ai/knowledge/documents/{document_id}
```

该能力用于读取 `document:*` 文档清单及解析统计，不得把 `doc:*` 向量分块数量误当作文档数量。

### 9.3 分块分页与上下文

Java 面向前端的证据接口：

```http
GET /api/manuals/{manualId}/versions/{version}/chunks?type=text&page=1&page_size=20
GET /api/manuals/{manualId}/versions/{version}/chunks?type=table&page=1&page_size=20
GET /api/manuals/{manualId}/versions/{version}/chunks/{chunkId}
```

Python 可提供相应知识读取能力供 Java 调用：

```http
GET /ai/knowledge/documents/{document_id}/chunks?type=text&page=1&page_size=20
GET /ai/knowledge/documents/{document_id}/chunks/{chunk_id}
```

要求：

- 读取时必须按 `document_id` 和允许的知识 `chunk_type` 过滤，不得扫描返回全部 `doc:*` 数据。
- 支持按页码与分块序号稳定排序，不使用相关性检索结果冒充顺序浏览结果。
- `page_size` 必须设置合理上限。
- 响应不得包含 Redis 中的向量二进制字段。
- 单块详情支持返回同页或前后相邻片段，便于审批人员核验上下文。

### 9.4 表格结构化返回

`DocumentParserTool` 已能提取表格行列；新增索引数据必须由 Python 在分块元数据中保存 `metadata.table_rows`。Java 聚合响应可按业务接口风格映射为 `tableRows`，使证据页不依赖前端拆解 Markdown 字符串：

```json
{
  "id": "6e68c6ed:00:tbl:0030",
  "chunkType": "table",
  "page": 22,
  "caption": "第22页表格",
  "content": "表格：第22页表格\n序号 | 料件名称 | 数量 | 备注\n...",
  "tableRows": [
    ["序号", "料件名称", "数量", "备注"],
    ["1", "M6×30 六角法兰面螺栓（环保彩锌）", "2", ""]
  ]
}
```

历史数据若不存在结构化行列，可返回原 `content` 并显式标记结构化内容不可用，不强制在本阶段重建全部历史文档。

### 9.5 原 PDF 与文件定位

原 PDF 证据查看必须满足：

- 私有 PDF 不转换成永久公开链接。
- `ManualVersion` 持久化 `storageBackend`、`sourceBucket`、`sourceObjectKey` 和 `contentType` 等稳定定位信息。
- MinIO 模式下，Java 在用户请求时生成短期有效访问地址，或以 `application/pdf` 代理返回文件流。
- `local` 模式下继续通过受控静态资源或文件流提供访问，不依赖 MinIO 字段。
- 图片与 PDF 上传和返回时明确设置可预览的 MIME 类型。
- 计划证据固定索引版本和页码，手册换版后历史计划仍可打开原依据。

### 9.6 搜索兼容与知识隔离

现有接口：

```http
POST /ai/knowledge/search
```

继续用于语义检索，并保留既有响应结构。新增闭环不得要求已有调用方立即迁移。

知识隔离规则：

- `doc:*` 当前可能混合手册知识分块和会话事实，本阶段所有证据浏览与计划检索必须限定 `document_id`、索引版本及知识分块类型。
- Python 计划生成仅检索 Java 下发 `manualScope` 中的服务版本，不引用未就绪、未批准或会话事实内容。
- 现有搜索结果对图片摘要的兼容表达继续保留：图片摘要可按 `chunk_type = "image"` 返回，同时以 `source_chunk_type = "image_summary"` 标识来源，避免已有前端展示逻辑失效。
- 中期演进可将手册知识分块与事实记忆拆为独立前缀和独立索引，但不作为本阶段交付阻塞项。

## 10. 现有 AI 会话图片兼容修复

任务主流程使用 `imageIds` 和 Java 下发的受控图片地址。与此同时，现有 AI 对话链路存在 Java 单个 `url` 与 Python `images` 列表字段不一致的问题，可能导致演示中的故障图片未真正进入分析流程。

前置修复要求：

| 项目 | 设计规则 |
|---|---|
| Java 对 Python 的图片请求 | 统一传递 `images: [url]`，支持一张或多张图片 |
| 访问控制 | 图片 URL 为短期受控地址，不暴露永久私有文件路径 |
| 兼容目标 | 保证旧 AI 对话场景仍可使用图片检索，同时新任务流程以任务证据为准 |
| 验证方式 | 补充上传图片后调用流式会话接口的联调测试，验证 Python 实际接收到图片列表 |

该兼容修复不改变“业务状态仅由任务与审批实体控制”的原则。

## 11. 失败处理、降级与一致性

| 故障场景 | 系统行为 |
|---|---|
| 手册上传失败 | 不创建可引用版本，记录失败原因并允许重试 |
| Python 索引失败 | 候选版本标记失败；已有服务版本继续可用 |
| 索引结果迟到或重复 | 按 `jobId` 幂等，非当前候选版本不得切换服务版本 |
| 计划生成超时 | 不创建待审批计划，任务保留可重试状态 |
| 无有效手册证据 | 不生成可审批草案，不以纯模型知识形成作业指令 |
| 图谱无路径 | 手册支持的建议可保留，但对应部分标记未验证 |
| 图谱接口暂不可用 | 可生成带警告草案，审批人必须显式确认 |
| Java 保存计划失败 | 不推进任务至待审批，按 `requestId` 支持安全重试 |
| 高风险步骤未确认 | 拒绝提交该步骤执行结果 |
| 验收未通过 | 不关闭任务，允许新计划版本继续处理 |
| 案例草稿被驳回 | 不写入图谱和正式知识索引 |

## 12. 权限、安全与审计

### 12.1 第一阶段角色责任

| 角色 | 能力边界 |
|---|---|
| 执行人员 | 创建任务、上传现场证据、提交步骤结果和验收材料 |
| 审批人员 | 审批整份计划、确认警告、高风险步骤执行前确认 |
| 案例审核人员 | 修正并批准或驳回案例草稿 |
| 内部服务 | Python 可处理索引、检索和生成，但不得直接改变审批结论或案例发布状态 |

### 12.2 安全底线

| 风险 | 防护要求 |
|---|---|
| 配置凭据泄露 | 移除已跟踪真实凭据并完成轮换，改用安全配置注入 |
| PDF 与现场图片暴露 | 保持私有存储，由 Java 鉴权后提供短期访问或代理读取 |
| Python 越权引用手册 | 仅允许使用 Java 指定的 `manualScope` 与具体服务版本 |
| AI 编造作业依据 | 无有效手册证据时不得形成可审批计划 |
| 未审核案例污染知识库 | 仅审核通过案例可发布到图谱和索引 |
| 内部 API 被滥用 | 内网隔离并增加服务令牌或请求签名、追踪标识和访问日志 |

### 12.3 审计事件

以下行为必须记录操作者、时间、关联请求号、结果和相关实体版本：

- 手册上传、换版、索引成功切换、索引失败和清理。
- 计划生成、版本替换、审批、驳回和警告确认。
- 图谱验证成功、未找到路径或服务降级。
- 高风险步骤确认、执行结果和证据上传。
- 复测验收与任务关闭。
- 案例草稿修改、审核、发布和索引触发。
- 内部服务失败、重试和幂等命中。

## 13. 测试门禁与验收

### 13.1 Java 测试重点

| 类别 | 覆盖内容 |
|---|---|
| 状态机测试 | 手册版本切换、计划审批、步骤执行、验收和案例发布 |
| 风险规则测试 | 固定规则提升风险等级并拦截未确认执行 |
| API 测试 | 手册证据、任务、计划、审批、执行、验收及案例接口 |
| 幂等与乱序测试 | 重复或迟到索引结果不会覆盖错误服务版本 |
| 权限测试 | 未授权原文访问、审批或案例发布被拒绝 |
| 文件访问测试 | 私有 PDF 短期访问、本地存储模式和 MIME 类型行为正确 |

### 13.2 Python 测试重点

| 类别 | 覆盖内容 |
|---|---|
| 索引测试 | `manualId + indexVersion` 隔离，成功与失败状态正确回传 |
| 知识隔离测试 | 手册检索不混入会话事实 |
| 证据读取测试 | 列表、详情、分块分页、上下文和表格结构化读取正确 |
| 搜索兼容测试 | 现有 `/ai/knowledge/search` 行为不被破坏 |
| 图谱工具测试 | Agent 调用 Java 权威接口，并正确处理无路径与不可用状态 |
| 图片兼容测试 | `images` 列表链路能接收现场图片并参与处理 |
| 错误分支测试 | 搜索失败不产生二次异常，失败状态可稳定返回 |

### 13.3 闭环联调场景

1. 上传首次手册，索引成功，浏览手册分块与原 PDF，创建任务，生成计划，审批，执行，验收，审核案例并发布。
2. 已有可用手册时上传新版本，新索引失败，旧版仍可浏览并用于生成带证据计划。
3. 表格证据命中后，审批页可显示结构化表格和对应 PDF 页。
4. 图谱接口不可用时生成带警告草案，未确认警告不得批准，确认后可继续。
5. 高风险步骤未确认时无法提交执行结果。
6. 验收未通过时任务不关闭，可生成新计划版本处理。
7. 案例草稿被驳回时，不进入图谱和正式知识索引。
8. 现有知识搜索调用仍可返回已入库数据，且图片链路完成字段兼容。

## 15. 第一阶段验收标准

- Java 能管理手册服务版本与候选索引任务，新版本失败不影响旧版引用。
- 审批人员能够查询可用手册、浏览文本与结构化表格证据，并从引用定位至正确原 PDF 页。
- 计划生成仅引用 Java 指定的 `ready` 手册版本；没有有效证据时不形成可审批计划。
- 图谱验证只认可真实关系路径；图谱不可用时的草案明确提示并要求人工确认。
- 检修任务具备创建、计划审批、步骤执行、高风险确认、复测验收和关闭的完整状态链。
- 已批准计划的证据、风险确认和审批轨迹可审计，计划开始执行后不可被原地改写。
- 已验收任务仅生成待审核案例草稿，未批准案例不会进入正式图谱或知识索引。
- PDF 与现场图片以受控方式访问，文件定位与 MIME 信息足以支持稳定预览。
- 现有 `/ai/knowledge/search` 行为保持可用，AI 对话图片输入字段完成兼容。
- README、Schema 与演示说明反映最终落地接口与完整业务闭环。
