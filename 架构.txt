# FixAgent AI 模块架构

## 1. 项目概述

**本项目是 FixAgent 系统的 AI 推理引擎，专注于设备故障诊断推理、知识检索和维修作业指引生成。**

- **赛题**：第十四届中国软件杯 A1 赛题——"基于多模态大模型技术的设备检修知识检索与作业系统"
- **目标平台**：4核 8GB 龙芯虚拟机（LoongArch + 麒麟OS），需轻量、高效
- **职责边界**：Python 只负责 AI 推理（诊断、检索、指引、记忆），业务数据管理（CRUD、文件存储、会话管理）由 Java 后端统一负责
- **交互方式**：Java 后端通过 HTTP 调用 Python AI 模块，AI 模块通过 HTTP 调用外部服务

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| 边界清晰 | Python 只负责 AI 推理，不碰业务数据 |
| 松耦合 | AI 模块通过 HTTP 接口与 Java 后端交互 |
| 单 Agent 统一处理 | FixAgent 通过 ReAct 循环自主决定工具调用，无需意图路由 |
| 零 LLM 审核 | ReviewAgent 用 3 层确定性校验替代 LLM 自我审查 |
| 轻量部署 | 原生 Python 实现，不引入重型框架（LangGraph 等），适配龙芯 4核 8GB |
| 可追溯 | ReAct trace 全程可记录，支持事后审核 |
| 单例模式 | 各服务/Agent 均采用惰性单例，首次调用时初始化 |

## 5. 模块设计

### 5.1 agents/ — AI 推理层

#### BaseAgent (`base_agent.py`)

Agent 抽象基类，定义统一接口和模板方法。

**核心模型**：
- `AgentInput`：统一输入（user_message, session_id, images, context, conversation_history）
- `AgentOutput`：统一输出（agent_name, message, tools_used, metadata, latency_ms）

**提供的执行模式**：

| 方法 | 用途 | 使用者 |
|------|------|--------|
| `run()` | 基础 LLM 对话 | MemoryAgent, RealtimeMemoryAgent |
| `run_with_react()` | ReAct 工具调用循环 | FixAgent（非流式） |
| `run_with_react_stream()` | ReAct + SSE 流式 | FixAgent（流式） |
| `run_stream()` | 基础流式对话 | 暂无直接使用 |

**上下文注入机制**：`_build_messages()` 自动将 context 中的 `previous_summary`、`relevant_facts`、`user_preferences`、`session_preferences`、`unresolved_items` 注入 system prompt，实现无状态请求的有状态上下文。

---

#### FixAgent (`fix_agent.py`)

统一诊断 Agent，持有全部工具，通过 ReAct 循环自主决策。

- **执行入口**：`run_with_react()` / `run_with_react_stream()`
- **持有工具**：`knowledge_retrieval`、`graph_query_diagnosis_path`、`graph_search_devices`
- **工具调用策略**（在 system prompt 中定义）：
  - 简单知识查询 → knowledge_retrieval → 直接回答
  - 故障诊断 → knowledge_retrieval + graph_query_diagnosis_path → 综合分析
  - 维修指引 → knowledge_retrieval + graph_query_diagnosis_path → 标准化步骤
  - 不确定设备 → graph_search_devices → 确认后再诊断
- **回答格式**：诊断结论 + 操作步骤（含操作内容、所需工具、安全注意）
- **单例**：`get_fix_agent()`

---

#### ReviewAgent (`review_agent.py`)

3 层确定性校验管线，零 LLM 调用。仅第 3 层会修改输出内容。

**第 1 层：检索依据校验（`_GroundingCheck`）**
- 拆分回答为句子 → 识别事实性陈述 → 从 react_trace 收集检索证据
- 向量化句子和证据 → 计算余弦相似度矩阵
- 相似度 < 0.35 的句子标记为"未验证"
- 向量化失败或证据缺失时默认通过

**第 2 层：图谱路径校验（`_GraphCheck`）**
- 从回答中提取故障-方案对应关系
- 优先用 react_trace 中图谱查询结果做 O(1) 匹配
- 未命中时调用 `GraphService.fault_node_exists()` / `solution_node_exists()` 确认节点存在性
- Neo4j 不可用时用 trace 结果兜底

**第 3 层：安全规则引擎（`_SafetyCheck`）**
- 7 类危险操作关键词匹配（高压电气/高温/化学品/重物/旋转部件/压力容器/电池）
- 检查对应安全关键词是否存在
- 缺失则自动追加标准化安全警告文本

**内联标记**：`get_inline_markers()` 为流式输出提供标记位置，在未验证内容前插入 `⚠️[依据不足]` / `⚠️[图谱:xxx]` 标记。

---

#### MemoryAgent (`memory_agent.py`)

工作记忆整理，将多条原始对话压缩为结构化摘要。

- **执行入口**：`run()`（覆盖基类，实现 function calling 流程）
- **工具**：`search_similar_facts` — 批量搜索语义相似的历史事实
- **输出**：`MemorySummary` — new_facts / superseded_ids / updated_preferences / updated_unresolved / resolved_item_ids / brief_summary
- **自动入库**：提取的新事实自动向量化写入 Redis，带 `metadata.type="fact"` 标记
- **重试机制**：JSON 解析失败自动重试 1 次
- **触发条件**：Java 端对话数 >= 30 条时调用

---

#### RealtimeMemoryAgent (`realtime_memory_agent.py`)

实时记忆更新检测，每轮对话后异步调用。

- **检测范围**：仅检测"事实纠正"和"偏好变更"两类操作
- **执行模式**：纯判断（不用 function calling），直接 JSON 输出，< 3s 完成
- **偏好操作类型**：
  - `upsert`：新增或修改偏好
  - `delete`：撤销/删除偏好
- **事实纠正流程**：用 wrong_content 向量匹配旧事实 → 删除旧向量 → 写入正确事实 → 返回 superseded_fact_ids

---

### 5.2 services/ — 外部服务客户端层

#### LLMService (`llm_service.py`)

封装阿里云百炼 API。

- **模型**：qwen-plus
- **API 端点**：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- **核心方法**：
  - `chat()` — 基础对话（支持流式/非流式、response_format）
  - `chat_with_tools()` — 带工具调用的对话（自动处理 tool_call 循环，最大 10 轮）
- **客户端**：httpx.AsyncClient（连接池复用，60s 超时）
- **单例**：`get_llm_service()`

---

#### VectorService (`vector_service.py`)

基于 Redis Stack 的向量存储和检索。

- **索引名**：`knowledge_vectors`
- **向量维度**：1024（与 qwen2.5-vl-embedding 匹配）
- **距离度量**：COSINE
- **索引算法**：HNSW
- **核心方法**：
  - `add_vector()` / `add_vector_batch()` — 写入向量
  - `search()` — 原生向量 KNN 搜索
  - `search_by_text()` — 文本→向量→搜索（便捷方法）
  - `delete()` — 删除向量（通过删除 Redis Hash key）
- **过滤支持**：按 category（TAG）、tags（TAG）过滤
- **工具函数**：`build_redis_filter(category, tags)` — 构建 RediSearch 过滤表达式，供 API 层和工具层复用
- **索引自动管理**：启动时自动创建/迁移索引

---

#### GraphService (`graph_service.py`)

Neo4j 图数据库只读查询服务。

**图谱 Schema**：

```
(Device)-[:OWNS]->(Component)-[:CAUSES]->(Fault)-[:HAS_SOLUTION]->(Solution)
(Device)-[:HAS_FAULT]->(Fault)   ← 历史故障记录
```

**核心方法**：

| 方法 | 功能 |
|------|------|
| `find_devices()` | 按关键字模糊匹配设备 |
| `search_components_by_embedding()` | 向量匹配部件（Neo4j 向量索引） |
| `search_faults_by_embedding()` | 向量匹配故障（Neo4j 向量索引） |
| `fault_node_exists()` | 检查故障节点是否存在（模糊匹配） |
| `solution_node_exists()` | 检查方案节点是否存在（模糊匹配） |
| `find_diagnosis_paths()` | 5 分支诊断路径分页查询 |
| `find_components_by_device()` | 查询设备下的部件 |
| `find_faults_by_component()` | 查询部件关联的故障 |
| `find_solutions_by_fault()` | 查询故障的解决方案 |

**5 分支路由逻辑**：根据传入的 device_ids / component_ids / fault_ids 组合，自动选择最优 Cypher 查询分支。

---

#### KnowledgeService (`knowledge_service.py`)

知识导入编排服务，编排 `文档解析 → 向量化 → Redis 入库` 完整流程。

- **处理单元**：按 PDF 章节（"第X章"）分组处理
- **文本**：分块向量化（批量 25 条/批，百炼 API 限制），存入 Redis
- **表格**：转 markdown 文本后向量化入库
- **图片**：用图注文本向量化入库（metadata 记录本地路径）
- **单例**：`get_knowledge_service()`

---

### 5.3 tools/ — 工具执行层

#### BaseTool (`base_tool.py`)

工具抽象基类，模板方法模式。

- **核心模型**：`ToolResult`（success / data / error）、`ToolError`（code / message）、`ToolException`
- **模板方法**：`run()` 自动捕获 ToolException → ToolResult，未知异常 code="TOOL_ERROR"
- **OpenAI 兼容**：`to_openai_schema()` 生成 function calling 格式的工具定义

---

#### KnowledgeRetrievalTool (`knowledge_retrieval_tool.py`)

知识库向量检索工具。

- **支持模式**：纯文本检索 / 图文混合检索（文本 + 图片向量平均融合）
- **参数**：query, top_k, category, tags, image_urls
- **过滤**：支持 category 和 tags 的 RediSearch 过滤表达式

---

#### GraphQueryTool (`graph_query_tool.py`)

图谱诊断路径查询工具。

- **查询流程**：component_description → 向量匹配部件 → fault_description → 向量匹配故障 → 5 分支 Cypher 查询
- **返回**：每条路径含向量分数、历史记录标记、可读路径文本

#### GraphSearchDeviceTool (`graph_query_tool.py`)

图谱设备搜索工具，按关键字模糊匹配 Device 节点。

---

#### FactRetrievalTool (`fact_retrieval_tool.py`)

事实向量检索工具，MemoryAgent 专用。批量并发检索多条候选事实的相似历史事实。

---

#### DocumentParserTool (`document_tool.py`)

PDF 文档解析工具。

- **技术栈**：pdfplumber（文本+表格提取）+ PyMuPDF / fitz（图片提取，可选）
- **关键能力**：章节自动识别（"第X章"正则）、图注匹配、表格清理、CMYK→RGB 转换
- **远程文件**：HTTP URL 自动下载到临时目录

---

### 5.4 embeddings/ — 向量化层

#### TextEmbedding (`text_embedding.py`)

文本向量化，使用 `qwen2.5-vl-embedding` 模型，输出 1024 维。

- **缓存**：Redis 缓存（key: `emb:v2:{md5}`），TTL 24h
- **批量**：`embed_batch()` 自动分离缓存命中/未命中，减少 API 调用

#### ImageEmbedding (`image_embedding.py`)

图像向量化，使用同一 `qwen2.5-vl-embedding` 模型，输出 1024 维，确保图文统一语义空间。

- **缓存**：Redis 缓存（key: `img_emb:v2:{md5}`）
- **输入**：图片 URL 或 base64

#### MultimodalEmbedding (`multimodal_embedding.py`)

门面模式（Facade），对外提供统一入口，内部按输入类型分发给 TextEmbedding / ImageEmbedding。

---

### 5.5 schemas/ — 数据模型层

| 文件 | 内容 |
|------|------|

## 6. 核心数据流

### 6.1 故障诊断流程（/ai/chat）

```
用户: "电动机轴承过热是怎么回事？"

Java Backend
  ├── 从 Redis 获取 ChatMemory（会话历史）
  ├── 调用 /ai/memory/search_facts 检索相关历史事实
  ├── 组装 ChatRequest（message + session_id + conversation_history + context）
  └── POST /ai/chat

FixAgent.run_with_react()
  ├── _build_messages(): system prompt + context + history + user message
  ├── get_tools(): 注册 3 个工具
  └── chat_with_tools() ReAct 循环:
       ├── Thought: "需要查图谱理解因果链路"
       ├── Action: graph_query_diagnosis_path(fault_description="轴承过热")
       │   └── TextEmbedding.embed() → Neo4j 向量索引匹配
       │   └── 5分支 Cypher 查询诊断路径
       ├── Observation: 部件→故障→解决方案链路
       ├── Thought: "还需要检索维修手册"
       ├── Action: knowledge_retrieval(query="电动机轴承过热原因")
       │   └── TextEmbedding.embed() → Redis FT.SEARCH KNN
       ├── Observation: 检索到维修手册相关段落
       └── Answer: 综合诊断结论 + 维修步骤

ReviewAgent.review()
  ├── 第1层 _GroundingCheck: 向量相似度验证事实性陈述
  ├── 第2层 _GraphCheck: Neo4j Cypher 确认故障-方案对应关系
  └── 第3层 _SafetyCheck: 关键词匹配，缺失安全警告自动追加

ChatResponse → Java Backend
  { message, tools_used, verification, latency_ms }
```

### 6.2 流式对话流程（/ai/chat/stream）

```
采用「先缓冲再验证」策略：

SSE 事件序列:
  session_id → status(FixAgent分析中) → tool(工具调用) →
  status(验证中) → marker(⚠️未验证标记) → token(逐字) → ...
  verification(校验摘要) → done(耗时/工具)

关键点：
  - ReAct 阶段实时推送 status/tool 事件（展示进度）
  - token 先缓冲不发送，验证完成（~300ms）后才流式输出
  - 在未验证内容字符位置前内联插入 marker 事件
```

### 6.3 知识导入流程（/ai/knowledge/import）

```
Java 后端上传 PDF URL
  → KnowledgeService.import_document()
    → DocumentParserTool: 下载 → pdfplumber 逐页提取文本/表格 + PyMuPDF 提取图片
    → 按章节合并 → 文本分块（每批 25 条）→ TextEmbedding.embed_batch()
    → 表格转 markdown → TextEmbedding.embed()
    → 图片图注 → TextEmbedding.embed()
    → VectorService.add_vector_batch() → Redis
  → KnowledgeImportResponse（file_name, text/image/table count, process_time_ms）
```

### 6.4 记忆整理流程（/ai/memory/consolidate）

```
Java 后端检测对话数 >= 30
  → POST /ai/memory/consolidate（含旧偏好/旧未完成/上一轮摘要）
  → MemoryAgent.run()（function calling 模式）
    → LLM 读取对话 → 提取候选事实 → search_similar_facts 检索
    → 判断冲突 → 输出 MemorySummary JSON
    → 新事实自动向量化入库（Redis，metadata.type="fact"）
  → MemoryConsolidateResponse → Java 存储到 MySQL + 事实向量表
```

### 6.5 实时记忆更新流程（/ai/memory/realtime_update）

```
每轮对话完成后 Java 异步调用（不阻塞主对话流）
  → RealtimeMemoryAgent.run()
    → LLM 纯判断（仅看本轮消息，JSON 输出）
    → 如有事实纠正：
      → wrong_content 向量匹配 → 删除旧事实 → 写入纠正后事实
      → 返回 superseded_fact_ids 给 Java 标记旧记录
    → 如有偏好变更（upsert/delete）→ 返回 preference_changes 给 Java
  → 全程 < 3s
```

## 7. 数据模型

### 7.1 核心请求/响应

**ChatRequest**：session_id, message(max 50000), images(最多10张), stream, conversation_history, context

**ChatResponse**：session_id, message, tools_used, verification(3层校验结果), latency_ms

**KnowledgeImportRequest**：file_url, file_type(pdf), category, tags

**KnowledgeImportResponse**：file_name, total_pages, text_count, image_count, table_count, sections, process_time_ms

**MemoryConsolidateResponse**：session_id, summary(MemorySummary), original_count, consolidated_at

### 7.2 图谱数据模型（Neo4j）

```
Device(id, name, code, model, location, manufacturer)
  └─ OWNS →
Component(id, name, partNumber, specification, supplier, lifecycle, embedding)
  └─ CAUSES →
Fault(id, code, name, description, severity, category, embedding)
  └─ HAS_SOLUTION →
Solution(id, code, title, description, toolsRequired, estimatedTime, difficulty, verified)

Device ─ HAS_FAULT → Fault (历史故障记录)
```

### 7.3 向量存储模型（Redis）

```
key: doc:{doc_id}
fields: id, text, vector(FLOAT32[1024]), metadata(JSON), category(TAG), tags(TAG), created_at
索引: knowledge_vectors (HNSW, COSINE)
区分: metadata.type = "fact" | "text" | "table" | "image"
```

## 11. 关键设计决策

### 11.1 为什么不用 LangGraph

| 考量 | 结论 |
|------|------|
| 功能需求 | FixAgent 是线性 ReAct 循环，无复杂条件分支/并行/中断恢复需求 |
| 硬件约束 | 4核 8GB 龙芯 VM，LangGraph + LangChain 依赖链庞大 |
| 现有基础 | BaseAgent 已完整实现 ReAct + SSE + function calling |
| 维护成本 | 原生实现更可控，无框架升级兼容风险 |

### 11.2 为什么用 qwen2.5-vl-embedding 替代 text-embedding-v4

统一图文向量模型，文本和图片向量在同一语义空间（均为 1024 维），实现真正的图文混合检索（向量平均融合），无需额外的 CLIP 或跨模态对齐。

### 11.3 为什么不用 YOLO/SAM 做图片检测

- 赛题场景为"开放设备类型"（用户上传未知设备维修手册）
- YOLO 需要针对每种设备训练，无法覆盖所有设备类型
- Qwen-VL 多模态大模型零样本即可理解未知设备图片
- SAM 图像分割对文本检索流水线无直接价值

### 11.4 架构演进：从多 Agent 到单 FixAgent

旧架构：Orchestrator 意图路由 → 3 个子 Agent（检索/诊断/指引）

**问题**：
- 意图路由多一轮 LLM 调用（+1~2s 延迟）
- 子 Agent 工具集高度重叠，各自独立无法灵活组合
- 意图识别存在误分类风险

**新架构**：单一 FixAgent 持有全部工具，ReAct 循环中自主决策。ReviewAgent 用确定性校验替代 LLM 自我审查（零额外 LLM 调用）。

### 11.5 RealtimeMemoryAgent 与 MemoryAgent 的分工

| | RealtimeMemoryAgent | MemoryAgent |
|------|------|------|
| 触发时机 | 每轮对话后异步 | 对话数 >= 30 条 |
| 处理范围 | 仅纠正 + 偏好变更 | 完整整合（事实/偏好/待办/摘要） |
| 耗时 | < 3s | 40~60s |
| 架构 | 纯判断（无工具调用） | function calling |
| 关系 | 补充，不替代 | 主力整合 |
